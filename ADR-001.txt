ADR-001: USAR COMUNICAÇÃO SÍNCRONA (HTTP/REST) ENTRE MICROSSERVIÇOS

Status: ACEITO
Data: 19/04/2026
Autor: Equipe de Desenvolvimento

---

1. CONTEXTO

Em microsserviços, Order Service precisa consultar Product Service para validar produtos ao criar pedidos.

Alternativas principais:
- Síncrona: chama e espera resposta (como função normal)
- Assíncrona: publica evento e continua (como email)
- Banco compartilhado: ambos acessam mesmo DB

Cada opção tem prós e contras diferentes.


2. ALTERNATIVAS CONSIDERADAS

ALTERNATIVA A: Comunicação Síncrona (HTTP/REST) - ESCOLHIDA

Como funciona:
- Order Service faz GET /products para Product Service
- Espera resposta antes de continuar
- Resposta é imediata ou erro

Vantagens:
+ Simples: HTTP que todos conhecem
+ Debug fácil: testar com curl
+ Resposta rápida ao cliente
+ Sem infraestrutura extra

Desvantagens:
- Acoplamento: se Product cai, Order falha
- Cascata de falhas: atraso em Product afeta Order
- Latência maior: cada chamada adiciona tempo

Quando usar: respostas imediatas críticas, alta disponibilidade esperada, volume pequeno/médio


ALTERNATIVA B: Comunicação Assíncrona (Message Queue)

Como funciona:
- Order publica "pedido criado" em fila (RabbitMQ/Kafka)
- Order responde cliente imediatamente
- Product processa mensagem quando consegue

Vantagens:
+ Desacoplado: Product pode estar down
+ Resiliente: mensagens ficam na fila
+ Escalável: fila absorve picos

Desvantagens:
- Complexo: precisa infraestrutura de mensageria
- Debug difícil: mensagens invisíveis
- Latência maior: processamento não imediato

Quando usar: processamento em background, alta resiliência, volume alto


ALTERNATIVA C: Bancos de dados compartilhados

Como funciona:
- Ambos serviços acessam mesmo banco
- Order lê produtos diretamente da tabela

Vantagens:
+ Sem latência de rede
+ Dados sempre consistentes

Desvantagens:
- Viola microsserviços: não independentes
- Evolução difícil: mudança em produtos quebra Order
- Sem isolamento: falha em um afeta outro

NÃO recomendado: como monolito disfarçado.


3. DECISÃO

IMPLEMENTAR COMUNICAÇÃO SÍNCRONA VIA HTTP/REST

Justificativa:
Projeto educacional, simplicidade importa. HTTP/REST é padrão da indústria, fácil entender e debugar. Volume esperado pequeno, respostas imediatas fazem sentido. Aprendizado de HTTP vale mais que Kafka agora.


4. CONSEQUÊNCIAS

POSITIVAS:
✓ Simples e fácil entender
✓ Prototipagem rápida
✓ Debug com ferramentas comuns
✓ Custo operacional baixo

NEGATIVAS:
✗ Acoplamento temporal entre serviços
✗ Latência maior em picos
✗ Precisa timeout para evitar travamentos
✗ Em produção alto volume, seria problema

3. DECISÃO

Justificativa:
Projeto educacional, simplicidade. HTTP padrão indústria. Volume pequeno, respostas imediatas.

Positivas:
✓ Simples
✓ Fácil debug
✓ Prototipagem rápida

Negativas:
✗ Acoplamento
✗ Latência maior

4. CONSEQUÊNCIAS

4.1 Positivas
- Implementação rápida e direta ✓
- Código legível e mantível ✓
- Fácil testar com ferramentas HTTP ✓
- Debugging simplificado ✓
- Requisições são rastreáveis com Correlation IDs ✓

4.2 Negativas
- Se Product Service cair, Order Service não consegue criar pedidos ✗
- Latência de rede é adicionada ao tempo de criação de pedido ✗
- Sob alta carga, gargalo pode ser rate limiting de conexões ✗

4.3 Mitigações Implementadas
✓ Health checks: Ambos serviços expõem /health para monitoramento
✓ Correlation ID: Rastreamento de requisições entre serviços
✓ Logging estruturado: Facilita debugging de falhas
✓ Docker Compose: Garante ambos rodem juntos em desenvolvimento

4.4 Mitigações Futuras Recomendadas
- Implementar Circuit Breaker (biblioteca opossum)
- Retry com backoff exponencial (exponential backoff)
- Timeout configurável em requisições HTTP
- Implementar caching de produtos em Order Service
- Considerar migração para async se volume crescer


5. VALIDAÇÃO E TESTES

Teste manual realizado:
- Order Service consegue chamar Product Service via HTTP ✓
- Correlation ID é propagado corretamente ✓
- Ambos os serviços respondent dentro de latência aceitável ✓
- Health checks funcionam para ambos ✓

Comandos de teste:
```bash
# Testar Product Service
curl http://localhost:3001/products -H "X-Correlation-ID: test-123"

# Testar criação de Order
curl -X POST http://localhost:3002/orders \
  -H "Content-Type: application/json" \
  -H "X-Correlation-ID: test-123" \
  -d '{}'
```
