ADR-002: IMPLEMENTAR LOGGING ESTRUTURADO COM CORRELATION IDs

Status: ACEITO
Data: 24/04/2026
Autor: Equipe de Desenvolvimento

---

1. CONTEXTO

Requisição passa por UI Service → Demand Service → Deadline Checker Service → Email Service. Se erro em qualquer ponto, como correlacionar logs entre serviços?

Logs estão separados por container. Correlation ID único viaja por todos serviços para rastrear fluxo completo.

2. ALTERNATIVAS CONSIDERADAS

ALTERNATIVA A: Logging estruturado com Correlation ID - ESCOLHIDA

Como funciona:
- Gera UUID único por requisição inicial
- Propaga via header X-Correlation-ID
- Logs JSON incluem correlationId, service, method, url, timestamp

Exemplo log:
{
  "service": "demand-service",
  "correlationId": "abc123",
  "method": "POST",
  "url": "/demands",
  "timestamp": "2026-04-24T16:00:00.000Z"
}

Vantagens:
✓ Rastreabilidade completa do fluxo
✓ JSON estruturado para ferramentas
✓ Sem ferramentas extras necessárias
✓ Escalável para ELK/Datadog futuramente

Desvantagens:
- Logs stdout (perdem se container morre)
- Disciplina para sempre propagar header

ALTERNATIVA B: Logging centralizado custom

Criar microsserviço para logs centralizados.

Vantagens:
+ Logs centralizados desde início
+ Dashboard visual possível

Desvantagens:
- Mais complexidade desnecessária
- Overkill para projeto educacional

NÃO escolhido.

ALTERNATIVA C: Logging simples (console.log plain text)

Apenas console.log sem estrutura.

Vantagens:
+ Fácil implementar

Desvantagens:
- Impossível correlacionar entre serviços
- Sem contexto temporal/serviço

NÃO recomendado.

ALTERNATIVA D: Biblioteca logging (Winston, Pino)

Usar Winston para logging avançado.

Vantagens:
+ Profissional
+ Múltiplos outputs/transports

Desvantagens:
- Mais dependências
- Curva aprendizado desnecessária

NÃO escolhido: console.log + JSON suficiente.

3. DECISÃO

USAR LOGGING ESTRUTURADO COM CORRELATION IDs

Justificativa:
Padrão da indústria. Simples, escalável. Aprender Correlation IDs agora economiza tempo em projetos futuros.

4. CONSEQUÊNCIAS

Positivas:
- Rastreamento completo de requisições
- Debug facilitado entre serviços
- Preparado para observabilidade avançada

Negativas:
- Overhead mínimo de gerar UUID
- Disciplina para propagar headers

Mitigações:
- Middleware automático gera/propaga correlationId
- Logs incluem timestamp para ordenação
- Docker Compose facilita coleta de logs

5. IMPLEMENTAÇÃO

Middleware Express:

app.use((req, res, next) => {
    req.correlationId = req.headers['x-correlation-id'] || uuidv4();
    res.set('x-correlation-id', req.correlationId);

    console.log(JSON.stringify({
        service: "demand-service",
        correlationId: req.correlationId,
        method: req.method,
        url: req.url,
        timestamp: new Date().toISOString()
    }));

    next();
});

Ao chamar serviço:
axios.get('http://demand-service:3001/demands', {
    headers: { 'x-correlation-id': req.correlationId }
});

6. VALIDAÇÃO E TESTES

Testes realizados:
✓ Correlation ID gerado se não fornecido
✓ Propagado entre serviços
✓ Logs JSON válidos
✓ Header retornado

Teste manual:
curl -H "X-Correlation-ID: meu-id-teste" http://localhost:3001/demands
Procurar logs com "meu-id-teste" em todos serviços.
