ADR-003: USAR DOCKER COMPOSE PARA ORQUESTRAÇÃO LOCAL E TESTES

Status: ACEITO
Data: 19/04/2026
Autor: Equipe de Desenvolvimento

---

1. CONTEXTO

Dois serviços separados, como rodá-los juntos consistentemente? Como garantir que alguém execute tudo com um comando?

Requisitos: um comando inicia tudo, serviços comunicam, portas mapeadas, rede isolada, fácil debug, funciona em qualquer máquina.


2. ALTERNATIVAS CONSIDERADAS

ALTERNATIVA A: Docker Compose - ESCOLHIDA

Arquivo docker-compose.yml define todos serviços em YAML.

Um comando: docker compose up

Vantagens:
✓ Simples: um arquivo YAML
✓ Um comando inicia tudo
✓ DNS automático: serviços se encontram por nome
✓ Rede isolada segura
✓ Reproduzível: mesma config em dev/test
✓ Rápido: prototipagem em minutos
✓ Demonstração: funciona primeira tentativa

Desvantagens:
- Não production-grade: um host, sem auto-healing
- Sem orquestração avançada
- Sem load balancing


ALTERNATIVA B: Kubernetes (K8s)

Manifests YAML complexos para Pods, Services, Deployments.

Vantagens:
+ Orquestração profissional com auto-healing
+ Auto-scaling
+ Load balancing nativo
+ Rolling updates sem downtime

Desvantagens:
- Curva aprendizado alta (50+ páginas docs)
- Overkill para projeto educacional
- Setup complexo: Kind/Minikube/kubectl
- Tempo setup: 30-60 min apenas instalar

NÃO prático: projeto para aprender arquitetura, não Kubernetes.
- Impossível demonstrar em 2 minutos (precisa 15 min só de startup)

NÃO é prático: projeto é para aprender arquitetura, não Kubernetes.


ALTERNATIVA C: Docker run manual

Executar containers manualmente com docker run.

Vantagens:
+ Controle baixo nível
+ Aprende Docker profundamente

Desvantagens:
- Tedioso: múltiplos comandos complexos
- Propenso erros: vírgula errada quebra
- Impossível manter: esquece comandos
- Impossível demonstrar: 5 min typing com risco
- Documentação? "copy/paste isso"

NÃO prático: inviável para apresentação.


ALTERNATIVA D: Makefile ou shell script

Usar Makefile ou script bash para automatizar.

Vantagens:
+ Melhor que docker run manual
+ Documentado em arquivo

Desvantagens:
- Windows: Makefile não funciona nativo
- Cross-platform: bash não funciona Windows
- Ainda wrapper: docker run por trás
- Não melhora reproducibilidade

NÃO primário: Docker Compose resolve melhor.


3. DECISÃO

USAR DOCKER COMPOSE PARA DESENVOLVIMENTO E TESTES

Justificativa:
Docker Compose é ferramenta certa para dev local. Simples e poderosa, grandes empresas usam docker-compose.yml como linguagem de definição de infraestrutura para dev environments.

Para produção, migra para Kubernetes. Mas para aprender arquitetura agora, Docker Compose é perfeito.


4. CONSEQUÊNCIAS

POSITIVAS:
✓ Experiência simples: docker compose up
✓ Reproduzível: mesma config qualquer máquina
✓ Rápido: prototipagem minutos
✓ Escalável para learning: adicionar serviços depois
✓ Excelente demonstração: funciona primeira tentativa
✓ Fácil reset: docker compose down && up

NEGATIVAS:
✗ Não production: sem auto-healing
✗ Sem auto-scaling
✗ Sem load balancing


5. IMPLEMENTAÇÃO

Arquivo docker-compose.yml:

version: '3.8'
services:
  product-service:
    build: ./product-service
    ports: ["3001:3001"]
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:3001/health"]

  order-service:
    build: ./order-service
    ports: ["3002:3002"]
    depends_on:
      product-service:
        condition: service_healthy

networks:
  microservices-network:
    driver: bridge

Comandos:
docker compose up
docker compose logs -f
docker compose down
docker compose down

# Remover volumes (reset completo)
docker compose down -v

# Rebuild das images
docker compose up --build
- Service discovery automático ✓
- Rede isolada e segura ✓

Exemplo: Subir tudo em 1 comando:
```bash
docker compose up --build
```

4.2 Negativas
- Não escalável para produção ✗
- Limitado a um único host ✗
- Sem auto-healing de containers ✗
- Sem load balancing ✗

4.3 Mitigações Implementadas
✓ docker-compose.yml bem estruturado e comentado
✓ Ambos os Dockerfiles seguem boas práticas
✓ Rede explícita (automática pero documentada)
✓ depends_on garante ordem de inicialização
✓ Health checks via GET /health

4.4 Mitigações Futuras Recomendadas
- Para produção: Migrar para Kubernetes
- Adicionar volume para persistência de dados
- Implementar reinicialização automática (restart policy)
- Adicionar limite de recursos (CPU, RAM)


5. IMPLEMENTAÇÃO TÉCNICA - docker-compose.yml

5.1 Estrutura atual:

```yaml
services:
  product-service:
    build: ./product-service      # Constrói imagen a partir do Dockerfile
    ports:
      - "3001:3001"               # Expõe porta externa:interna
                                  # Acesso: http://localhost:3001

  order-service:
    build: ./order-service
    ports:
      - "3002:3002"
    depends_on:
      - product-service           # Garante que product sobe primeiro
```

5.2 Como funciona:
```
docker compose up --build
    ↓
Cria rede isolada "projeto-microservices"
    ↓
Constrói image de product-service a partir de Dockerfile
    ↓
Inicia container product-service acessível em http://product-service:3001
    ↓
Constrói image de order-service
    ↓
Inicia container order-service acessível em http://order-service:3002
    ↓
    ✓ Ambos rodando e comunicando-se via hostname interno
```

5.3 Service Discovery automático:
No Docker Compose, services se encontram automaticamente pelo NOME:
- order-service pode chamar product-service assim:
  ```javascript
  axios.get('http://product-service:3001/products')
  ```
  
Isso funciona porque Docker Compose cria uma rede com DNS interno
que resolve nomes de serviços para IPs dos containers.


6. COMANDOS ESSENCIAIS

Para apresentação ao vivo:

# Subir tudo (build + start)
docker compose up --build

# Ver status dos containers
docker compose ps

# Ver logs em tempo real
docker compose logs -f

# Parar tudo
docker compose down

# Remover volumes (data)
docker compose down -v

# Reconstruir images
docker compose build --no-cache

# Executar comando em container
docker compose exec product-service bash

# Ver variáveis de ambiente
docker compose config

# Verificar health
curl http://localhost:3001/health
curl http://localhost:3002/health


7. ESTRUTURA DE DOCKERFILES

7.1 Padrão usado (multi-stage seria overkill):

```dockerfile
FROM node:18                       # Imagem base confiável

WORKDIR /app                       # Diretório de trabalho

COPY package*.json ./              # Copia package.json e package-lock.json
RUN npm install                    # Instala dependências

COPY . .                           # Copia código fonte

EXPOSE 3001                        # Documenta porta (informativo)

CMD ["node", "index.js"]           # Comando de inicialização
```

7.2 Alternativas consideradas:
- Multi-stage build: Não necessário (não há dependências de build)
- Docker slim: Overkill para projeto educacional
- Alpine linux: Trade-off: menos 20MB size mas mais complexo

Decisão: Usar node:18 base simples
