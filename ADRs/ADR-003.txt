ADR-003: USAR DOCKER COMPOSE PARA ORQUESTRACAO LOCAL E TESTES

Status: ACEITO
Data: 24/04/2026
Autor: Equipe de Desenvolvimento

1. Contexto

Temos quatro servicos separados: Demand Service, Deadline Checker Service, Email Service e UI Service.
Queremos rodar tudo junto com um unico comando e garantir que os servicos se encontrem.

2. Alternativas consideradas

Alternativa A: Docker Compose - escolhido
- Use um arquivo docker-compose.yml para definir todos os servicos.
- Um unico comando inicia tudo.
- Docker cria rede interna e resolve nomes de servicos automaticamente.

Vantagens:
- Facil de usar.
- Reproduzivel em qualquer maquina com Docker.
- Bom para desenvolvimento e demonstracao.

Desvantagens:
- Nao e solucao de producao completa.
- Um host apenas.
- Sem auto healing e sem load balancing avancado.
- Multiplas instancias do mesmo container causariam conflitos no banco de dados SQLite compartilhado (deadline-checker.db), ja que SQLite nao foi projetado para acesso concorrente em escala de producao.

Alternativa B: Kubernetes
- Muito mais complexo.
- Overkill para este projeto educacional.

Alternativa C: docker run manual
- Funciona, mas e dificil e propenso a erro.

Alternativa D: script ou Makefile
- Ajuda, mas ainda depende de docker run manual e pode funcionar mal no Windows.

3. Decisao

Usar Docker Compose para desenvolvimento local e testes.

4. Consequencias

Positivas:
- Um comando para subir tudo.
- Servicos se encontram por nome.
- Facil de reiniciar e mostrar para avaliadores.

Negativas:
- Nao e o mesmo que um ambiente de producao.
- Fica limitado a uma maquina.
- Nao permite escalabilidade horizontal de instancias.

Apesar da falta de escalabilidade e das limitacoes do SQLite com multiplas instancias, esta decisao foi apropriada para o escopo do projeto educacional e ambiental de desenvolvimento.

5. Mitigacao

Mitigacoes para limitacoes de producao:
- Para producao, migrar para Kubernetes com load balancing e StatefulSet para deadline-checker.
- Implementar retry logic com exponential backoff nas chamadas HTTP entre servicos para lidar com falhas temporarias.
- Usar volumes persistentes (PersistentVolume) para dados criticos ao inves de volumes locais do Docker Compose.
- Implementar health checks avancados (liveness e readiness probes) em cada container.
- Para multiplas instancias, substituir SQLite por PostgreSQL compartilhado entre replicas do deadline-checker.
