ADR-004: USAR SQLITE PARA PERSISTENCIA DE LOGS NO DEADLINE CHECKER SERVICE

Status: ACEITO
Data: 24/04/2026
Autor: Equipe de Desenvolvimento

1. Contexto

O Deadline Checker Service precisa guardar historico de verificacoes e alertas.
O armazenamento deve sobreviver a reinicios e ser facil de consultar.

2. Alternativas consideradas

Alternativa A: SQLite com better-sqlite3 - escolhido
- Usa um arquivo local dentro do container.
- Um volume Docker garante persistencia entre reinicios.
- Suporta SQL para consultas e transacoes.

Vantagens:
- Zero configuracao.
- Backup facil.
- Relacional, simples de usar.

Desvantagens:
- Concorrencia limitada.
- Dados locais ao container.

Alternativa B: PostgreSQL ou MySQL
- Mais poderoso, mas adiciona um container extra.
- Mais complexidade sem necessidade para logs simples.

Alternativa C: arquivos JSON ou CSV
- Muito facil, mas sem transacoes e ruim para consultas.

Alternativa D: Redis ou Memcached
- R·pido, mas dados podem ser volateis e pouco flexiveis.

3. Decisao

Usar SQLite com better-sqlite3 para guardar logs do Deadline Checker.

4. Consequencias

Positivas:
- Persistencia fiavel.
- Consultas SQL.
- Backup facil.

Negativas:
- Concorrencia limitada.
- Nao distribuido.

5. Implementacao

Schema:
CREATE TABLE check_logs (
    id TEXT PRIMARY KEY,
    demand_id TEXT,
    title TEXT,
    status TEXT,
    deadline TEXT,
    days_remaining INTEGER,
    alert_type TEXT,
    message TEXT,
    checked_at TEXT,
    correlation_id TEXT
);

CREATE TABLE alert_logs (
    id TEXT PRIMARY KEY,
    demand_id TEXT,
    type TEXT,
    message TEXT,
    sent_at TEXT,
    correlation_id TEXT
);

Docker Compose inclui o volume deadline-checker-data para armazenar o arquivo deadline-checker.db.
