ADR-004: USAR SQLITE PARA PERSISTENCIA DE LOGS NO DEADLINE CHECKER SERVICE

Status: ACEITO
Data: 24/04/2026
Autor: Equipe de Desenvolvimento

1. Contexto

O Deadline Checker Service precisa guardar historico de verificacoes e alertas.
O armazenamento deve sobreviver a reinicios e ser facil de consultar.

2. Alternativas consideradas

Alternativa A: SQLite com better-sqlite3 - escolhido
- Usa um arquivo local dentro do container.
- Um volume Docker garante persistencia entre reinicios.
- Suporta SQL para consultas e transacoes.

Vantagens:
- Zero configuracao.
- Backup facil.
- Relacional, simples de usar.

Desvantagens:
- Concorrencia limitada.
- Dados locais ao container.

Alternativa B: PostgreSQL ou MySQL
- Mais poderoso, mas adiciona um container extra.
- Mais complexidade sem necessidade para logs simples.

Alternativa C: arquivos JSON ou CSV
- Muito facil, mas sem transacoes e ruim para consultas.

Alternativa D: Redis ou Memcached
- R�pido, mas dados podem ser volateis e pouco flexiveis.

3. Decisao

Usar SQLite com better-sqlite3 para guardar dados do Demand Service e logs do Deadline Checker Service.

4. Consequencias

Positivas:
- Persistencia fiavel.
- Consultas SQL.
- Backup facil.

Negativas:
- Concorrencia limitada.
- Nao distribuido.

5. Mitigacao

Mitigacoes implementadas em desenvolvimento:
- ✅ SQLite com melhor-sqlite3 (single-threaded) sendo usado em ambos os servicos.
- ✅ Volumes Docker garantem persistencia de demands.db e deadline-checker.db entre reinicializacoes.
- ✅ Uma unica instancia de cada servico rodando (conforme esperado em desenvolvimento).

Mitigacoes propostas para producao (nao implementadas neste projeto educacional):
- Migrar para PostgreSQL ou MongoDB para suportar acesso concorrente de multiplas instancias.
- Implementar retry logic com timeout nas queries para evitar locks prolongados.
- Usar WAL (Write-Ahead Logging) mode do SQLite para melhorar concorrencia em ambiente local.
- Backup automatico dos arquivos .db diariamente para evitar perda de dados.
