ARQUITETURA DO SISTEMA DE GERENCIAMENTO DE DEMANDAS

Este documento descreve como o projeto esta organizado e como os servicos trabalham juntos.

1. Visao geral

O sistema tem quatro servicos principais:
- UI Service: pagina web que exibe demandas e logs.
- Demand Service: guarda demandas e permite cria-las, editar e apagar.
- Deadline Checker Service: verifica prazos, calcula dias uteis e envia alertas.
- Email Service: envia alertas de email quando necessario.

Cada servico e isolado e roda em seu proprio container.

2. Como os servicos se comunicam

A comunicacao e sincronica via HTTP/REST.
O Deadline Checker chama o Demand Service para pegar a lista de demandas.
O Deadline Checker chama o Email Service quando precisa enviar alerta.
A UI usa fetch para conversar com os outros servicos.

Todos os servicos usam JSON no corpo das requisicoes.

3. Bounded contexts

Demand Service: responsavel pelos dados das demandas.
- Campos: id, title, description, status, deadline, createdAt.
- Controle completo de CRUD.

Deadline Checker Service: responsavel por verificar prazos.
- Usa os dados do Demand Service.
- Salva logs de verificacao e alertas em SQLite.

Email Service: responsavel por enviar emails.
- Recebe pedidos de alerta do Deadline Checker.
- Usa configuracao SMTP para entregar mensagens.

UI Service: responsavel pela interface web.
- Serve HTML/CSS/JS para o navegador.
- Faz chamadas a cada servico para mostrar dados.

4. Observabilidade

Usamos logging estruturado com Correlation ID para rastrear uma operacao em varios servicos.
Cada request pode carregar um header X-Correlation-ID.

Health checks estao disponiveis em /health para cada servico.
Isso ajuda a saber se o container esta funcionando.

O Deadline Checker guarda logs em SQLite para poder consultar historico de verificacoes e alertas.

5. Seguranca basica

No ambiente de desenvolvimento usamos HTTP.
Para producao, seria necessario HTTPS ou mTLS.

A entrada de dados deve ser validada no backend.
O projeto ainda nao tem autenticacao ou autorizacao.

6. Padrőes usados

- Comunicacao: REST sincronico.
- Servicos stateless quando possivel.
- Persistencia de logs com SQLite e volume Docker.
- Orquestracao local com Docker Compose.

8. Como usar

Para rodar tudo localmente:
- docker compose up --build

Para ver se os servicos estao no ar:
- curl http://localhost:3001/health
- curl http://localhost:3002/health
- curl http://localhost:3003/health
- acesse http://localhost:3000 no navegador
